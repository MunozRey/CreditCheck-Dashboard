### BUG-C1 ‚Äî `applyTheme()` race condition: T.* mutated during render

**File**: `src/constants/themes.js`, line 73‚Äì76
**Severity**: CRITICAL
**Root cause**: `applyTheme()` calls `Object.assign(T, ...)` synchronously ‚Äî if a React render is in progress when the theme toggle fires, components mid-render will read partially-written `T` values (e.g. `T.blue` is updated but `T.navy` is not yet).

**Reproduction**: Rapidly toggle the theme toggle button while the AnalyticsTab chart is rendering (chart has many `T.*` references). Some chart bars may flash wrong colors.

**Root cause**: `Object.assign` on the shared mutable `T` object is not atomic from React's perspective. The proper fix is `ThemeContext.jsx`'s context-aware `T = THEMES[theme]` (which is already done), but the monolith and some extracted components still use the global `T` import from `themes.js` directly (`import { T } from '../constants/themes.js'`), bypassing context.

**Fix (do NOT apply without full ThemeContext migration)**:
```js
// In applyTheme(), batch the mutation using a microtask so it fires
// after the current React render cycle finishes:
export function applyTheme(themeName) {
  // CSS injection can happen immediately
  // T mutation deferred to avoid mid-render race
  Promise.resolve().then(() => {
    Object.assign(T, THEMES[themeName]);
    Object.assign(CAT_STYLE, _buildCatStyle());
    Object.assign(GRADE_STYLE, _buildGradeStyle());
    Object.assign(MODEL_COLORS, _buildModelColors());
  });
  // ... CSS injection stays synchronous
}
```
**Note**: This is partially mitigated by `ThemeContext.jsx` already providing `T = THEMES[theme]` to all extracted components. The race only affects components that still `import { T }` from themes.js directly. Full fix = complete the ThemeContext migration (CLEANUP_PLAN Rule 4).

### BUG-C2 ‚Äî `fetchLiveData` has no timeout ‚Äî hangs the UI indefinitely

**File**: `src/utils/fetchLiveData.js`, line 33
**Severity**: CRITICAL
**Root cause**: `fetch(LIVE_ENDPOINT)` has no `AbortController` / timeout. If the ibancheck.io server is slow or hangs without closing the connection, the `fetching` spinner in the navbar stays active forever and the `runFetch()` lock never releases. The 60-minute auto-refresh interval will then stack additional pending fetches.

**Exact fix**:
```js
async function fetchLiveData(processRows) {
  const controller = new AbortController();
  const timeout = setTimeout(() => controller.abort(), 15000); // 15s timeout
  let res;
  try {
    res = await fetch(LIVE_ENDPOINT, { method: "GET", signal: controller.signal });
  } catch (err) {
    clearTimeout(timeout);
    if (err.name === "AbortError") {
      const timeoutError = new Error("Request timed out after 15s");
      timeoutError.isTimeout = true;
      throw timeoutError;
    }
    const corsError = new Error(`Network error ‚Äî likely CORS: ${err.message}`);
    corsError.isCors = true;
    throw corsError;
  }
  clearTimeout(timeout);
  // ... rest unchanged
}
```

### BUG-H1 ‚Äî Stale banner shows wrong staleness when lead `created` dates are in a different timezone

**File**: `src/App.jsx`, lines 210‚Äì211
**Severity**: HIGH
**Root cause**: `staleDays` and `staleHours` are calculated by comparing `new Date()` (client local time) against `new Date(snapshotDate)` where `snapshotDate` is a bare `YYYY-MM-DD` string. The `Date` constructor parses ISO date strings as UTC midnight, while `new Date()` is local time. A dashboard user in UTC+2 at 01:00 will compute `snapshotDate` as yesterday even if the data was uploaded at 23:00 local time on the same calendar day.

```js
// Current (buggy):
const staleDays  = snapshotDate ? Math.floor((new Date()-new Date(snapshotDate))/86400000) : null;

// Fix: parse the date string as local midnight, not UTC midnight:
const staleDays = snapshotDate
  ? Math.floor((Date.now() - new Date(snapshotDate + "T00:00:00").getTime()) / 86400000)
  : null;
const staleHours = snapshotDate
  ? Math.floor((Date.now() - new Date(snapshotDate + "T00:00:00").getTime()) / 3600000)
  : null;
```

### BUG-M2 ‚Äî `LeadScoringTab` DTI sort accesses `.expenses` / `.income` without null guard

**File**: `src/tabs/LeadScoringTab.jsx`, line 64‚Äì66
**Severity**: MEDIUM
**Root cause**: `a.income > 0` is false for `null` or `undefined` (returns `false`, not an error) but `a.expenses / a.income` when `income` is the string `"0"` returns `Infinity`, which sorts erratically. More seriously, leads from the Pipedrive format have `income: undefined` ‚Äî `undefined > 0` is `false` so the fallback `99` is used, but `parseFloat(undefined)` returns `NaN` which would break the sort comparator if it weren't masked by the `> 0` guard.

**Applied fix** ‚Äî added explicit `parseFloat()` guards:
```js
const da = (a.income != null && parseFloat(a.income) > 0) ? parseFloat(a.expenses) / parseFloat(a.income) : 99;
const db = (b.income != null && parseFloat(b.income) > 0) ? parseFloat(b.expenses) / parseFloat(b.income) : 99;
```

**File**: `src/utils/xlsxParser.js`, lines 68‚Äì91
